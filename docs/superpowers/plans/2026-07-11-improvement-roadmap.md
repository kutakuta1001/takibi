# Takibi 機能改善ロードマップ（優先度別計画・2026-07-11）

策定プロセス: Claude（Fable 5）起案 → Codex（gpt-5.5）と4往復の批判・すり合わせ → 収束。
議論ログ: `drafts/2026-07-11-improvement-discussion.md`
性格: 優先度と受け入れ基準を定めた**ロードマップ級の計画**。各フェーズの着手時に
superpowers:writing-plans で実装計画（タスク・テスト・コード詳細）を別途作成する。

## 前提と設計原則（変更なし・全フェーズ共通）

- 「自分では行けない場所へ連れて行く」（写真は探検の到達点・人工物写り込み不可）
- ゲーミフィケーション追加禁止 / 「効果として気づかれたら強すぎる」
- 現在地: 3スポット + 体験連鎖 + presence 磨きまで完成。公開が未着手

## Phase R0: 公開ゲート（最優先・規模目安1日）

「壊れず始まる・壊れた時に説明できる」に絞った社内公開の最低ライン。

1. タイトルのローダー化 — campsite パノラマ ready で開始ボタン活性。読み込み中・失敗・
   再試行の表示。起動時の3パノラマ一括生成をやめ、**campsite のみ先行ロード**
2. riverside / snowfield は初回遷移時に生成・ロード（暗転中に未ロードなら待つ + 表示 +
   失敗時は出発地に戻る導線）
3. WebGL 初期化失敗時の可読フォールバック文言
4. AudioContext resume の成否確認と、無音になった場合の案内（現状 unlock() は成否を返さない）
5. visibilitychange 対応 — 裏タブで audio suspend・燃料減衰/抽出/時間帯の進行停止
6. キーボードでの開始・クレジット到達（a11y 最低限。Title の全面クリック依存を解消）
7. モバイル閲覧時の「PC推奨」ガード表示（対応しないことと壊れた画面を見せることは別）
8. 炎動画（1.6MB）の非ブロッキング先読み — 失敗時は薪・火の粉・音で焚き火が成立する
   フォールバック

実装時の技術要点（Codex 最終レビュー指摘の反映）:
- PanoScene を非同期ロード化し「loading / ready / failed」の状態を持たせる（現状は
  コンストラクタで即ロード・状態なし）
- SpotManager に遷移失敗状態を追加 — 目的地パノラマのロード失敗時は current を維持して
  出発地に留まる（テスト対象）
- Engine の WebGLRenderer 生成を try/catch でガードし、失敗を UI に伝える経路を作る

受け入れ基準: 初回ロード・WebGL/audio 失敗・タブ非表示・炎動画失敗・キーボード操作・
モバイル閲覧の各ケースで、破綻せず「読める案内」または「成立する最低体験」に到達できる。
テスト条件（Vitest）: (1) 遷移先ロード失敗時に current が維持され onApply が呼ばれない
(2) visibility 一時停止中に燃料減衰・抽出が進まない。

備考: 写真クレジットが公開画面から到達できること（CC BY-SA の帰属表示は公開条件。
現状はタイトルにボタンあり = 充足。R0 の変更で導線を壊さないこと）。

## Phase R1: 公開体裁と品質（R0 直後・公開実行を含む）

1. ホスティング実施 — 着手前に CEO へ公開先を確認（GitHub Pages / Cloudflare Pages / 社内サーバー）
2. OGP / favicon / タイトル文言
3. 音量スライダー（マスター音量・localStorage 保存）
4. 既知課題の小修正 — 星マスクの輝度ベース化・斧の造形改善
5. fps デバッグオーバーレイ（`?debug=1`: avg fps / p95 frame time / renderer.info）
   → 社内テスターに各スポット30秒の確認とログ共有を依頼（自動回収はしない）

受け入れ基準: 公開 URL で OGP・favicon・クレジット・音量保存・debug fps が確認でき、
3スポットを通しで回って視覚・音・操作に明確な破綻がない。

（並行タスク）E2 の写真調査のみ先行実施可 — コード変更と独立のため。

## Phase E1: 座る・飲む体験の統合（体験拡張の第一弾）

1. 座る演出の汎用化 — Cooking に閉じている座りシーケンスを SitSpot として切り出し、
   Cooking はその利用者にするリファクタリング
2. riverside / snowfield に「座って眺める」スポット（報酬なし・浸る行動の追加）
3. **山頂の一杯** — kettle が 'ready' のまま雪山で座ると sip・湯気・消費が加わる
   （基本は既存意味論の流用: 'ready' はグローバル保持、drinkCoffee() で empty に戻る。
   ただし「GameState 無改修」を制約にはしない — 後続の体験追加に耐える最小拡張は許容し、
   拡張時はテスト先行で行う）

受け入れ基準: 3スポットの座る導線が同じ作法で動き、kettle ready 時だけ「山頂の一杯」が
発火して消費状態まで反映される。

## Phase E2: 第4の「行けない場所」（写真先行）

1. 写真調査（R1 と並行可）: 谷底・ゴルジュのパノラマを CC0 → CC BY / CC BY-SA の順で探索
2. 見つからなければ代替第一候補: **森林限界上の稜線/岩稜**（snowfield と地続きの世界観・
   「足元が切れ落ちる」到達感・人工物なし写真の入手性が谷底より高い・音は既存流用可）
3. スポット実装はライセンス・解像度・画角・クレジット表記の確定後に着手

受け入れ基準: 採用写真のライセンスと帰属が明記され、第4スポットが既存スポットと同じ
遷移・音・表示品質で追加される。

## Phase T: 様子見（公開後の実データで再判定）

- モバイル本対応（タッチ見回し・性能・レイアウト）
- 実ユーザー fps の回収の仕組み（同意・バックエンドが要るため当面やらない）
- ロード進捗表示の精密化（パーセンテージ等）

受け入れ基準: 実機・公開後の反応で必要性が確認された項目だけを個別に着手判断する。

## やらないリスト（今回の議論で確定）

| 項目 | 理由 |
|---|---|
| 釣り | 釣果・待ち時間・成功判定・道具UIが自然発生し、ほぼ確実にゲーム化する |
| テント設営 | 固定配置でも作業感・所有の方向へ寄り「到達点を眺める」体験から外れる |
| 時間帯選択 UI | 設定UIが増えるほど静かな体験が操作パネル化する。必要なら ?debug から |
| 雨天（グレーディング単体） | 晴れ写真との不一致が「効果として気づかれる」。実写雨パノラマ入手時のみ再検討 |
| 「朝を迎える」演出 | 写真の影・太陽が固定で朝は表現できない。現行の夜明けほどけが限界であり十分 |
| Google マップ写真活用 | 規約リスク。狙い（実在感）は実写パノラマ方式で達成済み。**バックログ正式クローズ** |
| 実ユーザー fps 自動回収 | バックエンド・同意・ノイズ除去が必要で社内公開には過剰 |
