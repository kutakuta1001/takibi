# Takibi 機能改善提案 — Claude x Codex 議論ログ（2026-07-11）

目的: 今後の機能改善を優先度別に計画する（実装はしない）。Claude（Fable 5）が起案し、
Codex（gpt-5.5）と4往復で批判・すり合わせを行い、最終計画に収束させる。

## 前提: アプリの現在地

- 実写360°パノラマのキャンプ体験。3スポット =
  campsite（深い原生林・Poly Haven CC0）/ riverside（渓谷の滝・CC0）/ snowfield（スイス3000m級
  山頂 Piz d'Err・Wikimedia CC BY-SA 4.0・アプリ内クレジット画面で帰属表示）
- 体験連鎖: 伐採→薪→焚き火が育つ→川辺で水汲み→火にかけ抽出→座って飲む。
  ノルマ・ゲームオーバーなし。誘導はプロンプト文言のみ
- 音: 全て Web Audio 手続き合成（風・川・鳥・虫・焚き火・SFX）+ スポット別リバーブ +
  風の突風変調 + 遷移時の足音（草/岩/雪）
- 演出: 夕⇔夜グレーディング（10分周期）・降雪・白い息・視点の呼吸ゆらぎ・無操作時UI消灯
- 技術: Vanilla Three.js + TS + Vite、テスト35件、`base:'./'` で静的配信準備済み。公開は未実施
- 設計原則（CEO確定）: (1)「自分では行けない場所へ連れて行く」— 写真は探検の到達点、
  人工物写り込み不可 (2) ゲーミフィケーション（実績・カウンター・報酬）追加禁止
  (3)「効果として気づかれたら強すぎる」
- 既知課題: 実機fps未計測 / snowfield谷底遠景に小さな集落陰影 / 星が最上部樹冠にわずかに残る /
  斧の造形が簡素
- 対象: PCブラウザ優先（モバイル未対応）。公開は社内向けから

## Claude 起案: 改善候補と初期優先度

### 優先度: 高（次の1〜2フェーズ）

- **H1. 公開と公開体裁**: ホスティング（静的）+ ローディング体験（初回 12MB 超のパノラマ・
  動画の読み込み表示。現状は無表示）+ OGP/favicon/タイトル文言 + 简易エラーフォールバック
  （WebGL 不可環境の案内）
- **H2. 実機パフォーマンス計測と最適化**: fps 計測、パノラマの遅延ロード（現状3枚一括の可能性）、
  テクスチャメモリ、タブ非アクティブ時の音・ループ停止
- **H3. 山頂の一杯**: campsite で淹れたコーヒーを持ったまま雪山へ行くと「山頂で飲む」が
  できる（既存 GameState の kettle 状態を流用。体験連鎖が3スポットを縫い合わせ、
  雪山に目的が生まれる。実績・報酬は付けない）
- **H4. 音量設定**: 環境音・SFX のマスター音量スライダー（設定は localStorage）。
  ゲーミフィケーションではなく利用環境への配慮

### 優先度: 中

- **M1. 谷底スポット**: CEO の原点発言「深い原生林や谷の底、雪山」の3つ目。見上げる崖・
  頭上の細い空。写真選定が最大の難関（CC0/CC BY-SA で人工物なしの谷底パノラマ）
- **M2. 座って眺める行動の追加**: 丸太・岩・山頂の岩場に「座る」だけのホットスポット
  （コーヒーなしでも座れる。浸る行動の追加・実装は小さい）
- **M3. モバイル対応**: タッチ見回し・パフォーマンス調整・レイアウト
- **M4. 時間帯の手動選択**: 「夜だけ浸りたい」ニーズ向けに、タイトルかどこかで開始時間帯を
  選べる（周期は残す）

### 優先度: 低（アイデア保管）

- **L1. テント設営体験**: キャンプ実感の核ではあるが「建築ゲームではない」原則との距離感を
  要議論。1張り限定・配置固定なら原則内か
- **L2. 天候バリエーション（雨の日）**: グレーディング + 雨粒 + 雨音レイヤー。写真は晴れのまま
  なので違和感リスクあり
- **L3. 焚き火料理の変奏（マシュマロ等）**: 連鎖の追加。コーヒーの純度を薄めるリスク
- **L4. フォトグラメトリ 3D モデル（岩・切り株の前景配置）**: 既存バックログ
- **L5. 既知課題の小修正**: 星マスクの輝度ベース化・斧の造形改善
- **L6. 釣り（川辺）**: スコープ拡大が大きい。ゲーム化リスク

### クローズ提案

- 「Google マップ写真の活用」バックログ: 発案の狙い（実在感）は実写パノラマ方式で実現済み。
  正式クローズしてよいか

## Codex への質問（Round 1）

1. 上記の優先度分けは妥当か。特に「公開（H1/H2）を体験拡張（H3以降）より先に置く」判断への賛否
2. Web 公開前の品質チェックとして抜けている観点はないか（大容量アセットのロード体験、
   ブラウザ互換、WebAudio の自動再生制約、アクセシビリティ、エラー処理など、
   このコードベースを見た上で具体的に）
3. 写真の時刻が固定という制約下で「朝焼けを迎える体験」「雨の日」はグレーディングで
   成立すると思うか（技術的リアリティの評価）
4. このリストに無い改善で、既存コード資産（GameState/音合成/スポット構造）を活かして
   小さい工数で体験価値が大きいものがあれば提案してほしい
5. 逆に「やらない方がいい」と考えるものはどれか（原則: 癒し体験・ゲーム化しない）

# Round 1: Codex の応答（要点）

- 「公開を体験拡張より先に」に賛成。根拠: main.ts が3スポット分の PanoScene を起動時に生成し
  TextureLoader が即走る。ロード表示・失敗表示・遅延ロードなしの公開は初回体験が不安定
- 優先度の変更提案: H4（音量）は高のまま公開前バッチへ / **L5（星・斧の小修正）を高へ**（初回の
  没入感に直撃・工数小）/ **H3（山頂の一杯）は中へ**（公開ブロッカーでない。drinkCoffee() が即
  kettle=empty に戻す設計のため単純流用では済まない）/ M2（座る）は中の上でH3より堅い /
  **M4（時間帯選択）は低へ**（設定UIが増えるほど操作パネル化）
- 公開前チェックの抜け: LoadingManager/progress/error 未実装（panos 9.5MB+動画1.6MB+岩4.1MB）/
  WebGLRenderer 生成失敗時の catch なし / AudioEngine.unlock() の resume 成功確認なし（Safari系で
  「始まったが音がない」リスク）/ **visibilitychange 未対応**（裏タブで音・燃料減衰・抽出が進む）/
  a11y 最低限（Title を button 相当に・キーボード到達・WebGL不可時の可読テキスト）
- 朝焼け: グレーディングの延長では「夜がほどけて夕景に戻る」が限界。「朝」と言わないこと
- 雨: 晴れ写真+雨演出は不一致が「効果として気づかれる」側に倒れる。雨の実写がある場合のみ
- リスト外の提案: (1) 遷移クリック直後の目的地パノラマ先行ロード（工数小・効果大）
  (2) visibilitychange 対応（小・中〜大） (3) スポット別の「ただ眺める」視点誘導 —
  lookAt() と座り演出の薄い再利用（小〜中・中）
- やらない方がいい: 釣り（ほぼ確実にゲーム化）/ テント（作業感・所有の方向へ寄る）/
  公開前の M4 / Google マップバックログは正式クローズ賛成

# Round 2: Codex の応答（要点）

- **H3+M2 統合に賛成**。ただし「雪山に飲むホットスポットを置くだけ」は楽観的: 現在の座って飲むは
  Cooking.ts の fireKettleInteractable に閉じ、視線誘導も焚き火前提。**座る演出を汎用 SitSpot に
  切り出し、kettle==='ready' のときだけ sip/湯気/consume を足す共通化が前提**。優先度は中の上で妥当
- **Phase R は大きい。「R0: 壊れず始まる・壊れた時に説明できる」に絞った公開ゲートを提案**:
  - 最低ライン: campsite ロード状態表示 + ready まで開始不可 + 失敗時の可読エラーと再試行 /
    WebGL 失敗フォールバック / AudioContext resume 成否確認（現状 unlock() は成否を返さない）/
    visibilitychange（audio suspend + 燃料・抽出の進行停止）/ キーボード開始・クレジット到達 /
    **写真クレジットが公開画面から読めること（CC BY-SA のため体裁でなく公開条件）**
  - 公開を止めない: OGP/favicon / 音量スライダー / L5 小修正 / 完全な遅延ロード最適化 /
    fps 計測UI常設 / 進捗パーセンテージ精密化
- fps 計測: `?debug=1` の dev オーバーレイ（avg fps / p95 frame time / renderer.info）で十分。
  実ユーザー自動回収は過剰（社内テスターに ?debug=1 で見てもらいログ共有）
- タイトルローダー化に賛成。炎動画 1.5MB は開始ブロッカーにせず、先読み + 失敗時は
  薪・火の粉・音で成立させる
- モバイル: 低〜中へ下げてよいが、**「PC推奨」ガード表示だけは公開時に必要**（未対応と壊れた
  画面を見せるのは別）
- 第4の場所の代替候補: **森林限界上の稜線/岩稜**（snowfield と地続きの世界観・「足元が切れ
  落ちる」到達感・人工物なし写真が谷底より見つかりやすい・音は既存流用可）
